# Ролевая модель и доступы

## Роли в приложении

В Canno есть две служебные роли и один публичный пользовательский контур:

- **Администратор (`admin`)** — полный доступ к панели `/admin`.
- **Редактор (`editor`)** — доступ к управлению квестами без админ-метрик и системных настроек.
- **Участник** — не логинится в панель, работает по персональной ссылке `/play/<token>`.

## Матрица доступа

| Раздел/действие | Admin | Editor | Участник |
|---|:---:|:---:|:---:|
| Вход в `/admin/login` | ✅ | ❌ | ❌ |
| Вход в `/editor/login` | ❌ | ✅ | ❌ |
| Главная панель `/admin` | ✅ | ❌ | ❌ |
| Создание/редактирование квестов (`/admin/quest/*`) | ✅* | ✅ | ❌ |
| Импорт/экспорт квестов | ✅ | ❌ | ❌ |
| Метрики, аудит, архив запусков | ✅ | ❌ | ❌ |
| Прохождение этапов `/play/<token>` | ❌ | ❌ | ✅ |

\* Технически админ также имеет права редактора: проверка в коде допускает `admin` и `editor` для editor-маршрутов.

## Где задаются логины и пароли

Все учётные данные задаются через **переменные окружения**:

- `CANNO_ADMIN_USER` — логин администратора (по умолчанию `admin`).
- `CANNO_ADMIN_PASSWORD` — пароль администратора в открытом виде (для dev/тестов).
- `CANNO_ADMIN_PASSWORD_HASH` — bcrypt-хеш пароля администратора (предпочтительно для production).
- `CANNO_EDITOR_USER` — логин редактора (по умолчанию `editor`).
- `CANNO_EDITOR_PASSWORD` — пароль редактора в открытом виде (для dev/тестов).
- `CANNO_EDITOR_PASSWORD_HASH` — bcrypt-хеш пароля редактора (предпочтительно для production).

Рекомендуемый порядок:

1. Для production задавайте только `*_PASSWORD_HASH`.
2. `*_PASSWORD` используйте только локально и в CI для упрощения запуска.
3. Не коммитьте `.env` с реальными секретами.

## Как работает авторизация

1. Пользователь входит через `/admin/login` или `/editor/login`.
2. Приложение сравнивает введённый логин с `CANNO_ADMIN_USER`/`CANNO_EDITOR_USER`.
3. Пароль проверяется по хешу (или по паролю, если хеш не задан).
4. При успехе создаётся серверная сессия в таблице `admin_sessions` с ролью и временем жизни.
5. В cookie `canno_session` пишется `session_id` (HttpOnly, SameSite=Lax).

## Ограничения безопасности

- Есть rate-limit на попытки входа: `MAX_LOGIN_ATTEMPTS` за `LOGIN_WINDOW_SECONDS`.
- Есть rate-limit на ввод ответов участниками: `MAX_STEP_ATTEMPTS` за `STEP_ATTEMPT_WINDOW_SECONDS`.
- Попытки логина и важные действия пишутся в аудит (`audit_events`).

## Логика работы приложения (кратко)

1. **Организатор** (admin/editor) создаёт квест и этапы в панели.
2. **Администратор** может делать экспорт/импорт, смотреть метрики и аудит.
3. **Система** генерирует/хранит participant-токены для персональных ссылок.
4. **Участник** проходит этапы по паролям; сервис учитывает таймеры этапа и прогресс.
