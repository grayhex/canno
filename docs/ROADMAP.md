# Roadmap развития Canno Quest (после MVP)

Ниже — приоритизированный бэклог для перехода от MVP к production-версии.

## P0 — Критично перед первыми реальными запусками

Статус на 2026-05-01: ✅ закрыто (P0.1–P0.4).

1. **Авторизация и защита админки**
   - Ввести вход в `/admin` (минимум: логин/пароль, сессия, logout).
   - Хранить пароль в виде хэша (bcrypt/argon2), убрать hardcode секретов.
   - Добавить rate-limit на попытки входа.
   - Критерий готовности: анонимный пользователь не может просматривать `/admin` и генерировать ссылки.

2. **Валидация и безопасность пользовательского ввода**
   - Санитизация полей форм, ограничение длины и формата `password`, `title`, `prompt`.
   - Базовая защита от brute-force для ввода пароля этапа (например, ограничение по IP/token).
   - Критерий готовности: некорректные данные не ломают приложение и не попадают в БД без проверки.

3. **Надежность хранения данных**
   - Добавить миграции схемы БД (Alembic или собственный migration runner).
   - Ввести резервное копирование SQLite-файла и инструкцию восстановления.
   - Критерий готовности: любое изменение схемы воспроизводимо в CI и на проде.

4. **Обработка ошибок и журналирование** ✅
   - Централизованный logging (INFO/WARN/ERROR) вместо print.
   - Кастомные страницы 4xx/5xx и безопасные сообщения без утечки деталей.
   - Логи покрывают ключевые действия админа (вход/выход/создание ссылки) и критичные события участников (невалидные token, rate-limit, ошибки пароля, истечение таймеров, завершение квеста).
   - Критерий готовности: ключевые действия админа и ошибки видны в логах.

## P1 — Функции, без которых трудно масштабироваться

Статус на 2026-05-01: ✅ P1.5–P1.8 закрыты.

5. **CRUD квестов и этапов в админке** ✅
   - Создание/редактирование/удаление квеста.
   - Управление этапами (порядок, пароль, лимит времени, текст подсказки).
   - Переключатель `active` и безопасная деактивация активного квеста.
   - Критерий готовности: новый квест полностью собирается в UI без ручного SQL.

6. **Улучшение модели участника**
   - Статусы участника (`new`, `in_progress`, `locked`, `completed`).
   - Ресет прогресса по участнику из админки.
   - Массовая генерация ссылок (пакетами) + экспорт CSV.
   - Критерий готовности: оператор может администрировать группу участников за 1–2 минуты.

7. **Наблюдаемость и метрики**
   - Метрики: стартов квеста, % завершения, среднее время прохождения, конверсия по этапам.
   - Отчет в админке за период (день/неделя/месяц).
   - Критерий готовности: можно увидеть, на каком этапе чаще всего «застревают» участники.

8. **Тестирование** ✅
   - Unit-тесты для логики таймеров, блокировок, переходов этапов.
   - Интеграционные тесты HTTP-эндпоинтов (`/play/<token>`, `/admin`).
   - Минимальный smoke-тест при старте приложения.
   - Критерий готовности: CI блокирует merge при падении тестов.

## P2 — Инфраструктура и эксплуатация

Статус на 2026-05-01: ✅ P2.9–P2.11 закрыты.

9. **Контейнеризация и окружения**
   - Dockerfile + docker-compose для локального/серверного запуска.
   - Разделение конфигураций: dev/stage/prod через env-переменные.
   - Критерий готовности: `docker compose up` поднимает приложение одной командой.

10. **Переход на Postgres** ✅
   - Подготовлены инфраструктурные настройки переключения БД через env-переменные и compose-профиль `postgres`.
   - Для стендов доступен запуск контейнера Postgres и подключение через `CANNO_DATABASE_URL`.
   - Критерий готовности: окружение готово к эксплуатации на Postgres-конфигурации.

11. **CI/CD** ✅
   - Подключен GitHub Actions workflow с автозапуском тестов и проверкой сборки Docker-образа.
   - Критерий готовности: каждый коммит автоматически проверяется и собирается.

## P3 — UX и продуктовые улучшения

12. **UX участника** ✅
   - Явный таймер на экране этапа и предупреждение «мало времени».
   - Более дружелюбные ошибки и подсказки без спойлеров.
   - Адаптация под мобильные устройства (крупные поля, кнопки, читаемость).

13. **Гибкая игровая логика** ✅
   - Нелинейные ветки (разные следующие шаги по ответу).
   - Несколько попыток с пенальти по времени.
   - Локализация (RU/EN) для международных запусков.

14. **Операционные инструменты** ✅
   - Импорт/экспорт квестов (JSON).
   - Версионирование сценариев квеста.
   - Архив завершенных запусков.

## Definition of Done для каждой задачи

- Есть критерий приемки и чек-лист ручной проверки.
- Есть тест(ы) или обоснование, почему тест не нужен.
- Обновлена документация (`README.md`/`docs/ROADMAP.md`).
- Нет регрессий по базовому сценарию: генерация ссылки → прохождение всех этапов → финиш.
